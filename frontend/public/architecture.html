Why This Creates Privacy
👤
Alice deposits 1 ALGO
On-chain sees: "Someone deposited 1 ALGO. Here's commitment 0x7a3f..."
👥
9 others deposit 1 ALGO
Pool now has 10 commitments. All look identical — same denomination, different hashes.
❓
Alice withdraws to Bob
On-chain sees: "Someone proved they own A commitment. Bob gets 1 ALGO." Which of the 10? Nobody knows.
Privacy = anonymity set. With 10 deposits, there's a 1/10 chance of guessing which deposit belongs to Alice. With 100, it's 1/100. The more deposits, the stronger the privacy.

System Overview — All Layers

Layer 1 — UI
React Frontend (TransactionFlow.tsx)
User selects tier, enters password, clicks Deposit/Withdraw. Manages tab state, shows progress, handles wallet connection via @txnlab/use-wallet-react.
Layer 2 — Crypto
Key Derivation & Note Management (privacy.ts)
Derives master key from wallet signature + password. Generates deterministic notes (secret, nullifier, commitment). Encrypts/decrypts note storage. This is where the privacy math happens.
Layer 3 — ZK Proofs
Circuit Proving (snarkjs + circom WASM)
Downloads 131 MB PLONK zkey from R2 CDN. Runs snarkjs.plonk.fullProve() in browser. Produces proof + public signals. Proof says "I know a secret for a commitment in this tree" without revealing which one.
Layer 4 — Transaction
Transaction Building (useTransaction.ts + plonkVerifierLsig.ts)
Encodes proof into 4 LogicSig transactions (PLONK verification split across 4 txns due to AVM opcode budget). Adds payment txn + pool app call. Groups atomically. Wallet signs the payment/app call txns; LogicSig txns are self-signed.
Layer 5 — On-chain
Smart Contracts (TealScript → AVM)
Pool contract verifies proof passed LogicSig validation (checks LogicSig address matches known verifier). Stores commitment in Merkle tree box. Records nullifier to prevent double-spend. Sends/receives ALGO.
Layer 6 — Storage
Note Persistence (localStorage + sessionStorage)
Notes encrypted with AES-GCM using per-tab ephemeral key (sessionStorage). Backups encrypted with AES-256-GCM + PBKDF2 100k rounds. Deposit index counter prevents duplicate commitments across tabs.
Layer 7 — Network
Relayer Network (Cloudflare Workers)
2 independent relayers submit withdrawal txns so recipient address isn't linked to sender IP. Relayer sees: proof + recipient + amount. Relayer never sees: secret, nullifier, which commitment is being spent.

Layer 2 Deep Dive — Key Derivation & Note Creation

Master Key Derivation

1
Wallet Signature wallet
Wallet signs a fixed message: "2birds privacy pool key derivation"
signData("2birds privacy pool key derivation") → signature (64 bytes)
2
Password Input user
User enters password (min 8 chars). Combined with wallet signature.
input = signature + UTF8(password)
3
PBKDF2 Derivation crypto
100,000 iterations of PBKDF2-SHA256. Produces 256-bit master key. Intentionally slow to resist brute-force.
masterKey = PBKDF2(input, salt="2birds", iterations=100000, hash=SHA256) → bigint (mod BN254_R)

Note Derivation (deterministic from master key + index)

1
Claim Deposit Index atomic
Atomically reads and increments counter. Two tabs can't get the same index.
idx = claimNextDepositIndex() // reads counter, writes counter+1 in same call
2
Derive Secret & Nullifier deterministic
Derived from master key + index. Same key + index always produces same note — enables recovery.
secret = SHA256(masterKey + "secret" + idx) mod BN254_R nullifier = SHA256(masterKey + "nullifier" + idx) mod BN254_R
3
Compute Commitment MiMC hash
Commitment is what gets stored on-chain. It binds the note to a specific denomination.
commitment = MiMC(secret, nullifier, denomination) // 3-input MiMC sponge // commitment is public. secret + nullifier are private (only you know them). // To withdraw, you prove you know the secret+nullifier for some commitment in the tree.

What each piece does

Secret
Proves ownership. Only the depositor knows this value. Required in the ZK proof to spend.
→
Nullifier
Prevents double-spend. Hash is published when withdrawing. If it's already recorded, the spend is rejected.
→
Commitment
Public ID of the deposit. Stored as a leaf in the Merkle tree. Doesn't reveal secret or nullifier.
→
Denomination
Baked into the commitment hash. Proves the amount without revealing which deposit you're spending.

Layer 5 Deep Dive — Merkle Tree (On-chain State)

Each pool contract maintains a 20-level MiMC Merkle tree. Deposits add leaves. The root changes with every deposit. Roots are stored in a ring buffer (10,000 entries) so recent roots remain valid for in-flight withdrawals.

Cn = MiMC(secret, nullifier, denomination)    |    H(L,R) = MiMC(left, right)    |    20 levels = 220 = 1,048,576 max deposits

On-chain box storage

cmt{idx}
Commitment at leaf index. "cmt" + uint64(idx)
One per deposit.
root{slot}
Root at ring buffer slot. "root" + uint64(slot%10000)
Stores recent roots.
kr{hash}
Known root lookup. "kr" + root_hash
Fast O(1) root validation.
null{hash}
Spent nullifier. "null" + nullifier_hash
Prevents double-spend.

What happens when you deposit

1
Contract receives commitment
commitment = proof.publicSignals[0] // from ZK proof
2
Store in box at next_idx
box["cmt" + next_idx] = commitment next_idx += 1
3
Recompute Merkle root (contract-side)
// Walk up 20 levels from the new leaf, recomputing parent hashes // Store new root in ring buffer slot and knownRoots lookup

What happens when you withdraw

1
Contract receives nullifier_hash + root
nullifier_hash = proof.publicSignals[1] root = proof.publicSignals[2]
2
Check nullifier not spent
assert box["null" + nullifier_hash] does NOT exist // double-spend check
3
Check root is known
assert box["kr" + root] EXISTS // proves the commitment is in a valid tree state
4
Record nullifier + send funds
box["null" + nullifier_hash] = 1 // mark spent inner_txn: pay recipient denomination amount

Layer 3 Deep Dive — ZK Proof (What the Circuit Proves)

Deposit Circuit

// Public inputs (visible on-chain) commitment // the leaf to add to the tree denomination // 100000, 500000, or 1000000 (microAlgos) // Private inputs (known only to prover) secret // random scalar nullifier // random scalar // Circuit constraint assert commitment == MiMC(secret, nullifier, denomination) // Proves: "I know the preimage of this commitment, and it commits to this denomination"

Withdraw Circuit

// Public inputs nullifier_hash // MiMC(nullifier) — recorded on-chain to prevent reuse root // current Merkle root — proves commitment is in the tree recipient // who gets the ALGO (prevents front-running) denomination // amount to withdraw // Private inputs secret // the depositor's secret nullifier // the depositor's nullifier pathElements[20] // Merkle proof (sibling hashes at each level) pathIndices[20] // left/right at each level (0 or 1) // Circuit constraints assert nullifier_hash == MiMC(nullifier) commitment = MiMC(secret, nullifier, denomination) assert MerkleProof(commitment, pathElements, pathIndices) == root // Proves: "I know a secret+nullifier for SOME commitment in this tree" // WITHOUT revealing WHICH commitment. That's the privacy.

How the proof gets verified on Algorand

Browser
snarkjs generates PLONK proof (π) from circuit WASM + zkey
→
Encode
Proof encoded into 4 LogicSig program args (AVM budget: 20k opcodes each = 80k total)
→
LogicSig x4
TEAL program runs PLONK verification math. If proof invalid → txn rejected.
→
Pool Contract
Checks LogicSig address matches known verifier. Trusts the proof passed. Updates state.
Why 4 LogicSigs? PLONK verification requires ~70,000 AVM opcodes. Each LogicSig gets 20,000 budget. So the verification is split across 4 transactions in an atomic group. The pool contract checks that all 4 LogicSig addresses match the deterministic PLONK verifier address.

Layer 7 Deep Dive — Relayer Network

Why relayers exist

Without a relayer, the recipient wallet submits the withdrawal transaction. An observer sees: "address X received funds from the privacy pool." If X is your known address, your deposit is linked. The relayer breaks this link by submitting the transaction on your behalf.

What the relayer sees vs. doesn't see

Relayer SEES
• ZK proof bytes
• Recipient address
• Denomination amount
• Nullifier hash
• Your IP address
Relayer CANNOT see
• Your secret (in ZK proof)
• Your nullifier preimage
• Which commitment is yours
• Your deposit transaction
• Your wallet address

Withdrawal via relayer — step by step

1
Frontend picks random relayer network
relayer = RELAYERS[Math.floor(Math.random() * 2)] // Relayer 1: privacy-pool-relayer.mitchhall16.workers.dev // Relayer 2: privacy-pool-relayer-2.mitchhall16.workers.dev
2
Random jitter delay timing
await sleep(random(5000, 30000)) // 5-30 second random delay
3
POST proof to relayer network
POST /withdraw { proof, publicSignals, recipient, poolAppId }
4
Relayer builds & submits txn on-chain
Relayer constructs the atomic group (4 LogicSig + pool call), signs with its own key, submits to Algorand. Deducts 0.05 ALGO fee from withdrawal amount.
5
Relayer returns txId network
{ txId: "ABCD1234...", status: "confirmed" }

UI
1. User Action
Select tier (0.1/0.5/1.0) → Enter password → Click "Deposit"
Crypto
2. Key Derivation
masterKey = PBKDF2(walletSign("2birds...") + password) → note = deriveDeposit(masterKey, claimNextDepositIndex(), tier) → produces secret, nullifier, commitment
Timing
3. Batch Window
Wait until next 15-minute mark (:00/:15/:30/:45). Multiple deposits from different users land in the same window → harder to correlate.
ZK
4. Proof Generation
Download 131 MB PLONK zkey (cached after first download). snarkjs.plonk.fullProve(input, wasm, zkey) → proof π + public signals [commitment, denomination]
Transaction
5. Build Atomic Group
6 transactions in atomic group:
[0-3] 4× LogicSig txns — PLONK proof verification (each carries part of the proof in args)
[4] Payment txn — User sends denomination ALGO to pool contract address
[5] App call — Pool contract deposit(commitment) method
Wallet
6. Wallet Signing
Pera/Defly/Lute signs txns [4,5] (payment + app call). LogicSig txns [0-3] are self-signed by the compiled PLONK verifier program. 90-second timeout if wallet doesn't respond.
On-chain
7. Contract Execution
Pool contract checks:
• LogicSig addresses [0-3] match known PLONK verifier → proof is valid
• Payment amount == expected denomination
• Stores commitment at box["cmt" + next_idx]
• Recomputes Merkle root up 20 levels
• Stores new root in ring buffer box["root" + (next_idx % 10000)] and box["kr" + root]
• Increments next_idx
Storage
8. Persist Note
saveNote({secret, nullifier, commitment, denomination, leafIndex, appId}) → AES-GCM encrypted in localStorage with ephemeral per-tab key from sessionStorage

Full Private Send Flow — All Layers

UI
1. Enter Privacy Address
Recipient shares ALGO_ADDRESS:HPKE_VIEW_PUBKEY. The view key is a separate x25519 keypair derived from wallet signature.
Crypto
2. Derive New Note for Recipient
Sender creates a fresh note (random secret + nullifier, NOT from sender's master key). Note is meant for the recipient to later withdraw.
HPKE
3. Encrypt Note with Recipient's View Key
encryptedNote = HPKE.seal(recipientViewPubkey, JSON.stringify({secret, nullifier, denomination}))
Only the recipient can decrypt this with their view private key.
ZK
4. Generate Private Send Proof
Single proof covers: deposit of new commitment + binding to recipient address. Public signals include the new commitment and the recipient's Algorand address.
Transaction
5. Build Atomic Group
[0-3] 4× LogicSig (PLONK verify) + [4] Payment (denomination ALGO) + [5] App call with encrypted note in txn note field
On-chain
6. Contract Stores Commitment
Same as deposit — commitment added to Merkle tree. The encrypted note is in the transaction's note field (on-chain, but only decryptable by recipient).
Recipient
7. Recipient Scans & Decrypts
Recipient's client scans pool transactions → tries HPKE.open(viewPrivateKey, txn.note) on each
If decryption succeeds → note is theirs → save it → can withdraw later
The sender's identity is never linked to the recipient's. The on-chain transaction shows: "someone deposited into the pool with an encrypted note." Only the recipient can read it.

Layer 6 Deep Dive — Note Storage & Backup

In-browser storage (localStorage + sessionStorage)

1
Tab opens → generate ephemeral AES key
tabKey = crypto.getRandomValues(32 bytes) sessionStorage["privacy_tab_key"] = tabKey // dies when tab closes
2
Save note → encrypt with tab key
ciphertext = AES-GCM(tabKey, JSON.stringify(notes)) localStorage["privacy_notes_{wallet}"] = ciphertext
3
Load notes → decrypt with tab key
plaintext = AES-GCM-decrypt(tabKey, localStorage["privacy_notes_{wallet}"]) notes = JSON.parse(plaintext)
If another tab or XSS script reads localStorage, they get encrypted gibberish. The decryption key only exists in sessionStorage of the original tab. But if XSS runs IN the same tab, it can still access sessionStorage.

Backup export/import (AES-256-GCM + PBKDF2)

E
Export
salt = random(16 bytes) iv = random(12 bytes) key = PBKDF2(password, salt, 100000 iterations, SHA-256) → AES-256 key ciphertext = AES-GCM(key, iv, JSON.stringify(notes)) output = base64(salt + iv + ciphertext) → .enc file
I
Import
parse base64 → extract salt(16) + iv(12) + ciphertext key = PBKDF2(password, salt, 100000, SHA-256) plaintext = AES-GCM-decrypt(key, iv, ciphertext) // wrong password → throws notes = JSON.parse(plaintext) → merge with existing notes (skip duplicates by commitment) → validate each against chain: isNoteSpent() → remove spent ones

Security Layers — Full Overview

Timing Protection

  • Batch windows — deposits snap to 15-min marks
  • Operation cooldown — 2 min between ops
  • Withdrawal jitter — 5-30s random delay
  • Cluster warning — alert at 3+ ops/session

On-chain Verification

  • PLONK ZK proofs via LogicSig (4 txns)
  • Nullifier box prevents double-spend
  • Known root ring buffer (10,000 entries)
  • Pool min 5 deposits for privacy
  • Fixed denomination tiers only

Key & Note Security

  • Master key = PBKDF2(wallet_sig + password)
  • Notes AES-GCM encrypted per-tab
  • Backups AES-256-GCM + PBKDF2 100k
  • Deterministic recovery via master key
  • Atomic deposit index (no tab races)

Network Privacy

  • 2 relayers — random selection per op
  • Relayer never sees note secrets
  • HPKE encrypted notes (private send)
  • CSP headers block unauthorized scripts
  • SRI hashes on all bundled scripts

Atomicity & Recovery

  • Split/combine checkpoint recovery
  • Stale root auto-retry (3 attempts)
  • Imported notes validated vs chain
  • Wallet signing timeout (90s)
  • Error visibility (toast, not hidden modal)
